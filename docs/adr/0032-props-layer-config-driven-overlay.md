# ADR-0032: 縁側の小物（props）＝台帳駆動のオーバーレイ層（演出は汎用語彙のみ・renderer 非依存）

- ステータス: Accepted（Inc1 実装・同日 v2 化＝entity+component 台帳・poll 配信・narrate「茶々が知る」まで実装。点火イベント=Inc2 は設計のみ）
- 日付: 2026-07-18
- 関連: ADR-0019（意味 state＋config 駆動の差し替えアセット・本 ADR はその子）, ADR-0010（皮）, ADR-0012（実天気）, ADR-0028（実時刻の昼夜）, 原則#2（住人から外さない）・#4（config 主導）
- 出自: Backlog「小物演出」構想（7/5 ユーザー着想・生成AI 3案の①）＋「蚊取り線香を置きたい」（7/18 ユーザー・Gemini 生成画像持ち込み）

## 背景 / 課題

縁側に生活小物（蚊取り線香・お茶・おにぎり…）を置いて"生活の場"にしたい。ただし2つの増殖リスクがある:
1. **小物1個＝専用演出1個**の積み上げ（煙・湯気・ゆれ…が個別コード化→JS が肥大＝テンポつまみ増殖・Scheduler God 化と同じ道）。
2. 縁側が賑やかになりすぎて「ただ居る」静けさ（原則#2）を壊す。

## 決定

1. **台帳駆動**: 小物は `assets/props.json`（台帳）に「何を・どこに・いつ・どんな演出で」を列挙。コードは
   描き方だけを知る（ADR-0019 の分業）。差し替えは `ENGAWA_PROPS_CONFIG` / `engawa.json[assets].props_config`
   （sprite/scene と同列の皮・ADR-0010）。欠損は表示スキップ＝起動を止めない。
2. **表示条件は実日付の月ゲート**（`months`・省略=常時）＝実天気(0012)・実時刻(0028)に続く**「実日付＝季節」の
   環境反応**。蚊取り線香は 6〜9月だけ縁側に出る。LLM 非経由・トークン0（ニャー♥と同族のクライアント完結）。
3. **演出（effect）は汎用語彙のみ・小物専用コード禁止**: 今は `rise`（小さな四角の粒が立ちのぼる＝煙/湯気/香り。
   色・間隔・発生点は台帳のパラメタ）の1語だけ。**新しい小物は — 動くものでも — コード0行・台帳に1行**。
   新語彙（例: ゆれる）を足すのは**2個目の実需が来た時だけ**（ゲーム registry・客人スキルと同じ YAGNI 流儀）。
4. **静けさの上限**: 同時に出す小物は少数（目安2〜3個）・演出付きは1個まで。演出の速度等の**つまみはユーザー設定に
   出さない**（台帳内の固定値のみ＝つまみ増殖の教訓）。
5. **台帳は renderer 非依存の契約**: 今の描画は WebView の DOM 重ね（z=1・昼夜膜 z15-17 の下＝夜は小物も暮れる）だが、
   **最終形＝縁側の canvas シーン化（7/18 ユーザー着想・庭側ビュー構想の終着点）でも台帳ごと引っ越す**。
   rise の JS も引っ越し前提の小さな自己完結関数に留める。
6. 素材は Gemini 生成の同画風ピクセル絵（chacha/scene と同じ出自）を `assets/props/` に置く（構想①のライブラリ方式）。
   蚊取り線香は持ち込み PNG から**描き込みの煙を連結成分分解で除去**（煙は rise で焚くため二重になる）。

## 検討した代替案

- **背景 scene.png に焼き込む**: 動かせない・季節の出し入れ不可・皮差し替えと衝突。却下。
- **画像の煙を CSS で動かす（層分割/フィルタ）**: 絵を差し替えた瞬間に煙の位置と画風がズレる＝皮の思想と衝突。
  procedural（rise）なら**どの絵でも焚ける**。却下（rise 採用）。
- **小物ごとに専用演出を書く**: 増殖の道。却下（語彙制）。
- **LLM（茶々）が小物を選ぶ**: 構想①の本命だが、まず台帳と描画層＝土台だけ（Inc1）。種注入と同型の
  「prop タグ」経路は**実需が育ったら Inc2**（トリガー設計＝時刻/天気/アーク/LLM は Backlog の分かれ目）。

## 追記（2026-07-18・v2＝「増える前提」の再設計＋茶々が知る）

ユーザー要請「小物は増える前提で設計工夫を（デザインパターン適用・ゲーム世界の定石）」を受け、同日 v2 化:
- **Entity + Component**: 台帳エントリの性質を独立塊に分離（`place`/`when`/`effect`/`narrate`）。消費者（system）は
  自分のコンポーネントだけ読む＝View は place/effect・判定は when・**prompts は narrate**。
- **条件は Registry（Specification 縮小版）**: `when` の field→述語を `props._WHEN` に登録・全フィールド AND・
  未知フィールドは無視（前方互換）。新条件（hours/weather…）は述語1個の追加。
- **Flyweight＋Observer（既存 poll 相乗り）＝資産と状態の分離**: 資産（dataURI/place/effect）は起動時に全部注入、
  **「いま出ている集合」は poll が毎回配る**（daynight と同じ経路）→ 月替わりに再起動不要・時間帯ゲートも自然に動き、
  **Inc2「茶々が点け消しする」は active 集合の書き換えだけで届く**（JS 無改修）。表示は `.prop.on` のフェード。
- **Facade（単一正本）**: `props.catalog()` を views（描画）と prompts（語り）が共有＝食い違いを構造で防ぐ。
- **茶々が知る（案①実装）**: active な `narrate` を環境行に合流（`user_narration`＝ソロ／`ambient_line`＝room 全員）。
  文言は「縁側には**あなたが焚いた**蚊取り線香がある」＝*フィクション上は彼女が焚いた*ことにする（物語記述で
  エージェンシーを立てる・シミュレーションはしない）。天気と同じ「持たせるだけ」＝言い立てさせない（原則#2）。
- 採用しなかったもの: 本格 ECS・prop ごとのクラス階層・イベントバス（poll で足りる）。

## 追記（2026-07-18・codex diff レビュー4件の反映）

- **正規化層＝正本境界を強化**: `image` は空でない str のみ（数値/list は entity ごと破棄＝views の os.path が
  TypeError で `build_web_html` ごと落ちる穴【高】を根治）。数値（place/effect）は `_num` で有限数＋範囲クランプ
  （`period_ms` 下限 250ms＝負値で setInterval ほぼ0ms の DOM 連打【中】を防止）。views 側も try 全包み＋JS の
  `??`/`Math.max` で二重弁。`left_pct=0` 等の **0 は有効値**（`||`→`??`【低】）。
- **画像実在チェックをカタログへ**【中】: 画像ファイルが無い entity はカタログから落とす＝「画面に出せない物は
  世界にも無い」。views（描画）と prompts（narrate）が構造的に同じ集合を見る＝絵の無い小物を茶々が語らない。

## 備考（Open Questions）

- **Inc2 本命＝②点火イベント（茶々が焚く「ように見える」）**: ゲート遷移を Scheduler が検知→茶々の一言
  （「蚊ぁ増えてきたな、線香点けよか」）→その後に active 集合へ追加＝出現。中座（ADR-0027）と同じ
  「機械イベントを世界観に溶かす」手口。**active 集合の正本が純関数→Scheduler 状態に移り View ポート拡張が要る**
  （規模見積り済み＝A の約2倍）。③「LLM が本当に判断」は②との体験差が薄く費用対効果で当面見送り。
- その他 Inc2 候補: 時間帯ゲート（述語1個）／お茶・おにぎりのライブラリ拡充（同画風 Gemini 生成・narrate 付き）。
- canvas シーン化（庭側ビュー最終形）の際は、本台帳＋rise を entity/particle に移植（契約は不変）。
