# ADR-0011: イベントは「アーク（1〜Nフェーズ）」で表す（単発・箱庭・来訪を統一）

- ステータス: Accepted（**実装済み・P3.5 検証済み**）
- 日付: 2026-06-27
- 関連: ADR-0004, ADR-0005, ADR-0006, ADR-0008, ADR-0012, TECH_RULES §6, spec §11(events)

## 背景 / 課題
環境反応型の単体住人（ADR-0004）。だが自発のつぶやきは単調になる。変化を持たせるため、外から「出来事」を注入して茶々に反応させたい（TECH_RULES §6 が挙げる4イベント源 実環境・箱庭・話しかけ・来訪 のうち「箱庭」が未実装）。
さらに、単発の出来事だけでは薄い。時間をかけて展開する「起承転結」を持つ出来事（降ってきた→本降り→やんだ／雀が来た→ついばむ→飛び立つ）が欲しい。また来訪（ADR-0008）も「到来→滞在→辞去」という同型の時間構造を持つ。

## 決定
イベントを **「アーク（弧）＝1〜N フェーズの時系列」** として一般化する。
- フェーズは長命セッション（ADR-0005）に、ティックごとに narration として順次注入する。茶々は各フェーズに反応する／黙る（「……」可）を自由に選ぶ。
- **単発イベント = 1フェーズの退化したアーク**（猫が横切る 等）。
- **箱庭ストーリー = 起承転結を持つアーク**。フェーズ間に隙間（ふつうのつぶやき／沈黙）を挟み、2〜5分かけてほどける。
- **来訪（ADR-0008）= 中身が codex 往復になっている重いアーク**（ノック→入室→数往復→辞去）。
- スケジューラは毎ティック {進行中アークの次フェーズ / 新アーク開始 / ふつうのつぶやき or 沈黙} を重み付きで抽選。同時進行アークは原則1本（多くて2本）。
- アークは「Nティック内に結へ」の安全弁を持つ（詰まり防止）。

## 検討した代替案
- **単発イベントのみ（アーク無し）**: 実装は軽いが「ほどける」面白さ＝長命セッションの旨味を捨てる。却下。ただしアークは少数の味付けに留める。
- **来訪を別系統で実装**: P4を一品物で作ると二重実装。アークに乗せ、producer 差し替えで統一する。
- **全イベントをアーク化**: 過剰。単発で足りる出来事まで状態機にするのは YAGNI。1フェーズ=単発として包含する。

## 影響 / 帰結
- P3 機構をそのまま継承（フェーズ注入は kind="ambient"）→ cancel優先割り込み（ADR-0006）・QUIET_AFTER_USER・turn_lock が無改造で効く。
- 「箱庭」イベント源（TECH_RULES §6）が初めて具体化。位置づけは P3 と P4 の間（P3.5）。
- 来訪（ADR-0008）はこのアークの上に producer として乗る＝P4はアークに「往復する中身」を差すだけになる。
- アークの状態は spec の events テーブル（§11）が本来の永続化先。当面はメモリで持つ。

## 備考
- 増やすほど喋らせない。アークは**刺激の多様化**であって発話頻度の増加ではない（CLAUDE.md 原則：黙っていい・短くていい）。起承で黙り、転で初めて反応する、が自然。
- 天気を扱うアークには制約がある（実天気を偽れない）。その線引きは ADR-0012 で別途決める。
- フィール依存の機能。形を固める前に、実演可能な箱庭アーク（雀 等）を1本流して手応えを取る。
