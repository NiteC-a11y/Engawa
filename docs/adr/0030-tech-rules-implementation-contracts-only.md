# ADR-0030: TECH_RULES は固有の実装契約に限定する（決定の言い換えは書かない・0016 の refine）

- ステータス: Accepted（実施済み）
- 日付: 2026-07-14
- 関連: ADR-0016（文書の役割分担）, TECH_RULES, CLAUDE.md

## 背景 / 課題

7/14 の齟齬点検で、TECH_RULES.md の腐り方に明確なパターンが出た:

- **古くなっていた記述は、すべて「ADR/CLAUDE.md の決定を言い換えて写した部分」**だった。
  - §6「客人の出力はナレーション化（塀の向こうから声が…）」→ adr/0015 の部屋方式で置換済み（コードに該当文言なし）
  - §2「session/new は起動時1回」→ adr/0021（timeout 段階回復）・`/restart`・adr/0027（中座）の3経路で張り直すようになっていた
  - §1「天気は大阪 lat/lon 固定」→ 設定化済み（`ENGAWA_WEATHER_*`・既定が大阪なだけ）
- 一方、**TECH_RULES にしか書かれていない固有情報はすべて正確なまま**だった（§2 ワイヤ契約・request_permission の応答 JSON・§4 Windows の npx.cmd/taskkill/CREATE_NO_WINDOW 地雷）。

つまり腐るのは「決定のキャッシュ」であり、固有の実装契約は腐らない。決定を言い換えて持つ限り、ADR が増えるたび同期対象が増え、原則7（docs 齟齬点検）が開発とともに重くなる——0016 が spec v2 で却下した「二重管理＝同期ずれの再生産」を、TECH_RULES が節単位で再演していた。

## 決定

**各事実の正本を1箇所にする振り分けルール**を定め、TECH_RULES を刈り込む:

1. **経緯・却下理由** → `docs/adr/` のみ。
2. **全体像・現況・env つまみ一覧** → `CLAUDE.md` のみ。
3. **ワイヤ契約・OS 固有の落とし穴・越えてはならない境界・テストの回し方** → `TECH_RULES.md` のみ。
4. **決定そのものは TECH_RULES に書かない**＝必要なら1行＋ADR ポインタ。言い換えの写しを足したくなったら、それは ADR ポインタにする合図。

あわせて本 ADR の実施として、齟齬修正（上記3件＋欠落: agent ポート/openai backend の行・`ENGAWA_OPENAI_ALLOW_REMOTE` の課金防止ルール・timeout/bounded wait/-32603 再送）を反映し、§3/§7 の言い換え段落をポインタ化する。**§番号（1〜9）は温存**——`src/acp.py` のコメントや CLAUDE.md 原則6 が「TECH_RULES §2/§9」で参照しており、アンカーとして生きているため。

## 検討した代替案

- **TECH_RULES を削除して分配する**: 固有情報の行き先がない。CLAUDE.md は毎セッション context に載る正本で、Windows の spawn 手順や応答 JSON の形まで入れると肥大する。ADR は決定のジャンルで、レシピ集は収まりが悪い。参照も 39 箇所/16 ファイル（adr/0016・adr/0023・`src/acp.py`・README×2 等）にあり更新コストが利得に見合わない。却下。
- **現状維持で齟齬を都度直す**: 重複という構造原因を残したままでは、ADR が増えるたび点検が重くなり続ける。却下。

## 影響 / 帰結

- TECH_RULES が約半分に痩せ、同期対象が「固有の実装契約」だけになる＝原則7 の点検が軽くなる。
- 決定の詳細を知りたい読み手は ADR へ一段たどる（トレードオフとして受容。境界チェックリスト§8 とワイヤ契約は据え置きで即参照可）。
- 原則7 は継続する。本 ADR は点検対象を減らすのであって、点検を不要にはしない。
